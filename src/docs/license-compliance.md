# Third-Party License Compliance

This document is the canonical source for third-party dependency license compliance.

## Scope

- The required notice artifact is `THIRD_PARTY_NOTICES.txt` at the repository root.
- The About display artifact is `about.txt` at the repository root.
- Runtime notices cover:
  - production frontend dependencies reported by `corepack pnpm licenses list --prod --json`
  - Rust runtime dependencies resolved for the Windows release target with `cargo metadata --locked --filter-platform x86_64-pc-windows-msvc`
  - bundled icon resources from Google Material Symbols / Material Icons, recorded as Apache-2.0 assets
- Development and build tooling licenses are reviewed separately when those tools are redistributed with source, build tooling, or release packages.

## Notice Artifact Rules

- Regenerate `THIRD_PARTY_NOTICES.txt` after any dependency or lockfile change.
- The notice artifact must follow the compact release-notice format generated by `scripts/generate-third-party-notices.mjs`: corresponding source, dependency scope, detected license expressions, review notes, third-party component summary, grouped license sections, and license texts.
- The notice artifact must not include the project application version in its header.
- The component summary must include package names, versions, ecosystem, license expressions, package metadata when available, source URLs when available, modification status, and distribution scope.
- Standard license texts used by the dependency set must be included once in the generated notice artifact instead of duplicating the same license text under every component.
- Component-specific upstream NOTICE files, custom license files, unknown license expressions, or license text requirements not covered by the standard text section require explicit review before release.
- The app must display `about.txt` in the modal About flow. The About flow must not include a separate expandable Open Source Licenses section.
- Source and binary release artifacts must include `LICENSE` and `THIRD_PARTY_NOTICES.txt`.
- Windows app bundles must include `LICENSE`, `THIRD_PARTY_NOTICES.txt`, and `about.txt` as bundle resources or equivalent embedded text.
- Release checks must fail when the generated notice artifact is stale.

## License Review Rules

- MIT, BSD, ISC, Zlib, Apache-2.0, Unicode-3.0, and similar notice-style licenses require preserving copyright, permission, warranty, attribution, and notice text.
- Dual-licensed dependencies may be handled under the more practical compatible option unless the package metadata or package notice text says otherwise.
- MPL-2.0 dependencies require recipients to be informed where the MPL-covered source code is available.
- If an MPL-covered dependency file is modified, the modified MPL-covered source file must remain available under MPL-2.0 terms.
- New dependencies with GPL, AGPL, LGPL, custom, unknown, or missing license metadata require explicit review before release.

## Required Commands

- Update notices: `corepack pnpm licenses:generate`
- Verify notices: `corepack pnpm licenses:check`
- General project check must include the license check.
- Build release artifacts: `python build_release.py`
